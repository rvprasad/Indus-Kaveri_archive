This document provides normative way to do releases in project Indus.

  1. Each project should start with a version tag (version_0_0)
  2. After each "settled" change to the project, the project should be tagged via date tag.
  3. When the project reaches a state that warrants a new version (minor or major), the project should be tagged with a version tag.(version_0_1)
  4. Each time a project releases software, it should be tagged with a date tag (YYYYMMDD-HHMM) and a version tag that reflects the associated date tag (version_0_1-YYYYMMDD-HHMM).
  5. The name of every artifact in a release should reflect the version tag associated with it. (module-0.1-YYYYMMDD-HHMM)
  6. If an older version (version_0_0) is updated then it is released as module-0.0.1-YYYYMMDD-HHMM.
According to the above guidelines, all changes pertaining to an upcoming version will be released as enhancement to the current version. So, in case there are breaking API changes, the project should make an initial release of the upcoming version and release the artifacts as part of this new version. This system does not allow for alpha/beta releases. This is intentional.
